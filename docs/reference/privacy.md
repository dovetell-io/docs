---
title: Privacy and publication boundaries
description: Public docs should not expose private strategy, customer details, credentials, or unsupported claims.
---

Public docs must stay public-safe.

Before publishing, check for:

- credentials or tokens
- customer details
- private strategy
- proprietary implementation notes
- unsupported revenue, traction, or market claims
- internal-only repo names when they are not necessary
- draft material presented as canonical truth

## Promotion rule

Promote the reusable lesson, not the raw private context.

If a source document contains private or unresolved material, rewrite it for
the public audience instead of copying it directly.

## Paid docs boundary

Paid documentation can contain more implementation detail than public docs, but
it should still avoid private operating context unless the buyer is explicitly
licensed for that material.

Internal source notes should not leak into public or paid packages.
