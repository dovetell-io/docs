Introduction

What is dovetell?

dovetell is a context orchestration layer for AI-assisted software teams. Its north star: "I want to be asked fewer questions."

When teams use AI tools to build fast, context gets lost. Decisions made in chat threads, assumptions in someone's head, documentation that goes stale the moment work begins — all of it accumulates as context debt. dovetell exists to stop that accumulation.

One sentence: dovetell captures decisions, aligns intent, and keeps everyone from asking the same questions twice.

What dovetell is not

  • Not a documentation tool — those require humans to maintain them
  • Not a project management tool — no sprints, no deadlines
  • Not an LLM wrapper or chatbot
  • Not a code review tool
  • Not an AI-only product — it works without AI too

Three pillars

  • Capture — decisions and domain knowledge recorded as work happens
  • Align — context stays synchronized across code, docs, and team
  • Tell — the right context reaches the right person at the right moment

Introduction

Why it exists

Three failure modes that get worse as AI-assisted development accelerates.

01 — Tribal knowledge failure

Domain context lives in one person's head and never gets captured. When that person is heads-down or unavailable, the team stalls. Developers ask the same domain questions every sprint. The PM becomes a human search engine.

02 — Documentation drift

Governing docs are written once and go stale the moment work begins. Nobody trusts them, so nobody reads them. The gap between what the docs say and what the code does grows every sprint.

03 — Bi-directional blindness

Code evolves without informing docs. Docs change without reaching developers. The two never meet. As AI-assisted development accelerates, this gap widens faster than any team can manually close it.

The more AI assists in building, the more critical it becomes to maintain a clean, authoritative, living context layer — or outputs drift from intent faster than the team can catch them.

Why now

Teams are using Cursor, Claude Code, and GitHub Copilot to ship faster than ever. The tools handle execution. Nobody has solved what those tools need to know about the team to execute correctly. That's the gap dovetell fills.

Agent execution platforms give you three things: which model, which tools, which task. They don't give you domain context — what your team's definitions are, which decisions were made, what the architecture constraints are. That's dovetell's job.

The one-liner: Cursor handles the runtime. dovetell handles what the runtime needs to know about your team.

Why not just use Cursor / Claude Code / Cowork Dispatch?

These tools know how to execute. They don't know what your team decided last sprint, what your data retention policy is, or what "service boundary" means in your codebase. dovetell is the context layer those tools are missing.

The repo as source of truth

Context lives in .dovetell-context/ in the team's git repository. dovetell reads, renders, and writes back via a scoped token. No markdown content is ever stored in dovetell's database. The team owns the data. Always.

Introduction

Who can use it

dovetell is designed for AI-assisted teams but is not dependent on AI. Three modes, one product. Any team using git can benefit.

⚡
Mode 1 — Agentic
AI-native teams
Developer types @dovetell what is our data retention policy? inside Cursor or Claude Code. Gets the vetted answer with source and decision trace. Context assembled at query time via MCP server. This is the north star use case.
💬
Mode 2 — AI-assisted
LLM-using teams
Team uses Claude, ChatGPT, or any LLM but not in an agentic coding flow. They paste context from .dovetell-context/ into chat sessions. dovetell structures what they paste and keeps it vetted and current. The prompt library is the product for this tier.
🗂
Mode 3 — Git-native
Any team using git
Team uses git but minimal or no AI tooling. dovetell still works. The review workflow is a review workflow. The diff view is just a diff. The queue is a lightweight approval process. The vet history is an audit trail. No AI required at any point.

The git-native floor: ~100M developers use git. The product works for all of them without modification. AI is optional acceleration, not a dependency.

Why AI independence matters

For regulated industry buyers — healthcare, finance, defense — "we don't call AI models on your behalf" is a trust statement, not a limitation. Inference is local (Ollama) or BYOK. Your context never leaves your repo. dovetell is a structured markdown governance layer with a clean UI. The AI layer sits on top, optionally.

The positioning in one sentence

Don't position dovetell as an AI tool. Position it as a context governance layer that happens to make AI dramatically more useful. The AI story is the ceiling. The git story is the floor. The floor is much larger.

The Platform

Roles

Four roles. No custom roles. No inheritance. Each maps to accountability, not org chart position.

Owner
PM, team lead — accountable for context quality
✓ Vet ✓ Accept ✓ Add ✓ View ✓ Manage team
Approver
Senior engineer, tech lead, domain SME
✓ Vet ✓ Accept ✓ Add ✓ View — Manage team
Contributor
Developer, analyst — generates context
— Vet — Accept ✓ Add ✓ View — Manage team
Viewer
Stakeholder, ops lead, non-technical teammate
— Vet — Accept — Add ✓ View — Manage team

Design intent

Owner — one per project. This is the buyer persona. The person accountable for what the AI knows about the team.

Approver — the human in the review loop. Multiple per project, typically one per domain (architecture, data, policy). Makes context truth by accepting it.

Contributor — doesn't need to understand dovetell to participate. A developer imports a doc, it lands in the queue. The Contributor's job is to generate context, not govern it.

Viewer — the non-technical unlock. Sees the context base in plain English. No markdown visible. Can query, read, and monitor drift score. The same context that feeds a developer's @dovetell query renders for a PM, ops lead, or legal reviewer in plain language.

Why no custom roles: Custom roles create permission sprawl — the Teams/SharePoint failure mode. Four fixed roles maps to four types of accountability. If someone needs a new capability, you promote them. Simple is the feature.

The Platform

Review workflow

The human-in-the-loop gate. Nothing becomes truth without it. The mechanic is a GitHub pull request. The language is plain English.

Context change requests

When a context item is proposed or updated, it does not overwrite existing truth. It enters the queue as a proposed change — sitting alongside the current vetted version until an Approver acts.

GitHub conceptdovetell equivalent
BranchProposed change to a vetted item
Pull requestChange request in the review queue
ReviewerApprover
MergeAccept — item becomes new truth
Diff viewOld version vs. proposed, in plain English
Audit logVet history — who changed what, when

The non-technical reviewer UI

Viewers see a different surface than Approvers. No queue. No diff syntax. No markdown.

  • Plain English only — no raw markdown ever shown
  • One item at a time — not a dashboard
  • Two actions: "Looks right" or "Something's off"
  • Flags route to the Approver with an optional note
  • Mobile-first — ops leads and legal reviewers review on their phones

The same context base that feeds a developer's @dovetell query also surfaces to a non-technical teammate in plain English. dovetell is valuable beyond the engineering team.

Competitive position

dovetell sits between GitHub PR (rigorous, developer-only) and Google Docs suggestion mode (accessible, no audit trail). The only workflow built for both technical and non-technical reviewers with a full vet history.

The Platform

Capabilities

Five capabilities. The queue is the connective tissue — nothing becomes truth without it.

Context Console

The dashboard. A context health monitor, not a document editor. Opens at standup — sees the drift score, unvetted count, stale definitions, decisions captured this sprint. Four quadrant cards: Decisions, Assumptions, Definitions, Policies.

Unvetted Queue

The human-in-the-loop gate. Every automated update lands here first. Accept, edit, reject, or defer. Accepted items become vetted truth. This is where tribal knowledge stops being tribal. Includes diff view for proposed changes to existing vetted items.

Context Import

Drop in existing docs — markdown, Notion exports, Confluence exports, PDFs. dovetell extracts decisions, assumptions, and definitions via local or BYOK inference. Everything starts unvetted. First value in under an hour.

Context Query — MCP Server

dovetell as an MCP server. Developer types @dovetell what's our data retention policy? inside Cursor or Claude Code. Gets the team's vetted answer — sourced, with decision trace. "I want to be asked fewer questions" made literal.

Context Health Monitor

A drift score (0–100) per project. Surfaces stale items, unvetted queue length, and gaps flagged by the context analyzer. Formula TBD — not shipping until calibrated against real team data.

All capabilities are planned for Phase 2. None are currently live. The assessment and prompt library are Phase 1.

Reference

Definitions

Shared vocabulary. When a term appears in dovetell documentation or code, it means what is defined here.

Core concepts

context base — The living collection of decisions, assumptions, definitions, and policies that governs how a team's AI-assisted work should behave. Lives in .dovetell-context/ in the team's git repo.

context drift — The gap that grows between what a team's documents say and what the code or work actually does.

context orchestration — Assembling, aligning, and delivering the right context to the right person or tool at the right time.

tribal knowledge failure — Domain context that lives in one person's head and never gets captured.

unvetted — Status for any item captured automatically but not yet reviewed by a human. Cannot be queried as authoritative.

vetted — Status for any item reviewed and accepted by an Approver or Owner. Authoritative. Queryable.

decision trace — A record of what was decided, when, by whom, and why. Surfaced alongside query responses.

drift score — A metric (0–100) indicating how far a team's documented context has drifted from their actual work.

context change request — A proposed change to a vetted item. Does not overwrite truth until accepted. Equivalent to a git pull request for context.

renderer — dovetell's self-description. It reads, renders, and writes back to files in the team's repo. It does not own the data.

Four content types

decision — Something the team chose. "We standardize on OpenTelemetry for all new services." Has an owner, a date, and optionally a rationale.

definition — A term the team uses with a specific meaning. "Service boundary: the line between ingestion and processing layers."

assumption — Something the team takes as true but hasn't formally decided. "AWS is our primary cloud provider." May need to become a decision.

policy — A rule the team follows. "All data retained 180 days, then purged." Usually has compliance implications.

ID definitions

uid (userId) — Permanent per browser. Ties all projects for one person together.

pid (projectId) — One per project. Persists across retakes. Enables growth tracking.

aid (assessmentId) — Fresh per submission. Tracks individual runs within a project.

vid (versionId) — Which question set. Currently v01.

task-[8char] — Task GUID. 8-character nanoid, generated on task creation.

decision-[8char] — Decision GUID. 8-character hash. Enables cross-file linking between tasks.md, decisions.md, and other context files.

session hash — 8-character sha256-derived hash per Claude work session. Embedded in chat titles as (2a2d3774). Used as the source: field on Claude-generated tasks and decisions. Allows tracing any artifact back to its originating conversation.

Privacy

Plain English. No surprises.

What we collect

Required: email address only.

Optional: project name, role, team size, company, industry, primary AI tool, how you found us.

Automatically: assessment scores, section breakdown, raw answers, skip count, unique IDs (uid/pid/aid/vid), submission timestamp.

How we use it

  • To send you your results and unique tracking link
  • To enable progress tracking over time via your pid
  • The dovetell product team sees scores to improve the product
  • Occasional product updates — opt out any time by replying "unsubscribe"

What we don't do

  • Never sell your data to anyone
  • Never share with third parties or advertisers
  • No cookies (Plausible analytics is cookieless)
  • No tracking pixels

Platform data (Phase 2)

When the platform ships, dovetell will store identity and state only — user IDs, project IDs, queue state, vet history, sync timestamps, assessment scores. No markdown content is ever stored. Your decisions, definitions, assumptions, and policies stay in your repo. Always.

The principle: dovetell is a renderer, not an owner. We can't sell what we don't store.

Project name disclaimer

The project name field in the assessment gate says: "⚠ Don't include sensitive, confidential, or proprietary information here." dovetell is not a secure document store.
